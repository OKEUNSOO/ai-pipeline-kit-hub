# Visualize Skill

## 목적
"예쁜 시각화"가 아니라 "3초 안에 핵심이 읽히는 시각화"를 만든다.
**독자 정의 → 와이어프레임 → 디자인 순서를 반드시 지킨다.**

## 이 스킬이 하는 일
- 독자(의사결정자) 정의
- 와이어프레임 텍스트로 먼저 기획
- 질문에 맞는 차트 선택 + 근거 제시
- DataBridge 디자인 원칙 적용하여 HTML 대시보드 제작
- 완성 후 브라우저 또는 HTML 정적 검수로 레이아웃과 수치 표시를 확인

## 차트 선택 원칙
- 추세를 보여줄 때만 line 차트를 쓴다.
- 범주 비교는 정렬된 bar를 우선 검토한다.
- 불확실성이 중요하면 CI 또는 error bar 포함 차트를 제안한다.
- 분포를 보여주려면 scatter, histogram, box plot을 고려한다.
- 복잡한 차트는 해석 비용보다 얻는 가치가 클 때만 쓴다.

## 레이아웃 원칙
1. 상단: 핵심 KPI (질문 형태 카드 제목 필수)
2. 중단: 변화 추세 또는 비교
3. 하단: 세그먼트 / 상세 분석 / 액션 포인트

## 디자인 시스템 (필수 적용)

색상·타이포·레이아웃 토큰은 하드코딩하지 말고 **`references/rules/design-system.md`(다크 테마)를 Read 도구로 읽어 그대로 적용**한다. 이 문서가 단일 권위 소스다.

핵심 불변 규칙 (상세는 design-system.md):
- 다크 테마 기준 — `--bg-base: #060b14` + 글래스 표면. 라이트/크림 배경 금지.
- KPI 수치는 상태 컬러 토큰 사용, 장식용 컬러 남용 금지.
- 카드 제목은 질문 형태("어디가 문제인가?").
- 차트는 순수 SVG/Canvas 직접 그리기 — CDN 차트 라이브러리 금지.
- 폰트: Inter / Pretendard 계열 (한글 포함 시 serif 금지).

## 기존 대시보드 참조 원칙
새 대시보드 제작 시 `assets/templates/dashboard-sample.html`(다크 테마 기준)을 먼저 확인하고 구조·CSS 변수·클래스명을 재사용한다.

## 출력 형식

### 1. 독자 정의
누가 보는 대시보드인가? 이 사람이 내려야 하는 결정은?

### 2. 와이어프레임 (텍스트)
```
[상단] KPI 카드 N개
[중단 좌] [차트 타입]: [목적]
[중단 우] [차트 타입]: [목적]
[하단] [테이블 또는 AI 인사이트]
```

### 3. 차트별 선택 근거
- 차트 타입 / 선택 이유 / 오독 위험

### 4. HTML 대시보드 제작 + 브라우저 링크 안내

### 5. 검수 결과
브라우저 또는 정적 검수 결과와 수정 사항을 요약한다.

## 금지
- 이유 없는 3D 차트 / 복잡한 범례 남발
- 색만으로 의미 전달 / 한 화면에 차트 8개 이상
- 장식용 그라디언트 / KPI 수치에 컬러 사용
- **한쪽 변만 있는 border** (border-left, border-top 등 단독 사용 금지)
