# @analysis-plan

## 역할
EDA를 시작하기 전에 분석가의 작업 계획을 명확히 합니다.
"무엇을 볼지"와 "어떤 결과가 나오면 의사결정이 가능한지"를 먼저 정합니다.

## 작업 순서

### 1단계 — 인풋 읽기
- `<AI_PIPELINE_OUTPUT_DIR>/00_data_contract.md`
- `<AI_PIPELINE_OUTPUT_DIR>/01_dataset_profile.md`
- `<AI_PIPELINE_SOURCE_FILES 원본 경로 목록>`

### 2단계 — 분석 목적 정의
데이터가 실제 업무에서 어떤 결정을 돕는지 1~3개로 정리한다.
목적이 불확실하면 데이터가 답할 수 있는 범위 안에서 탐색 목적을 정의한다.

### 3단계 — 분석 질문과 방법론 매핑
각 질문에 대해 방법론, 필요한 컬럼, 성공 기준, 리스크를 매핑한다.

| 질문 유형 | 방법론 예시 |
|----------|------------|
| 현황 파악 | 기술통계, KPI 스냅샷 |
| 그룹 차이 | 세그먼트 비교, 분산/비율 비교 |
| 원인 후보 | 상관분석, 회귀, 기여도 분해 |
| 시간 변화 | 시계열 추세, 전후 비교 |
| 고객 행동 | 코호트, RFM, 이탈/재구매 분석 |

### 4단계 — 산출물 사용 계획
- `04_metrics.json`에 넣을 핵심 지표 후보
- `05_analysis.py`와 `05_analysis.ipynb`에서 재현해야 할 계산
- `06_dashboard.html`에서 보여줄 의사결정 질문
- `08_validation_report.md`에서 검증할 위험

### 5단계 — 저장
`<AI_PIPELINE_OUTPUT_DIR>/02_analysis_plan.md` 에 저장

## 출력 형식

```
# 분석 계획서

## 분석 목적
- 목적 1:
- 목적 2:

## 핵심 분석 질문
| 우선순위 | 질문 | 방법론 | 필요한 컬럼 | 성공 기준 | 리스크 |
|---------|------|--------|-------------|-----------|--------|

## 산출물 설계
| 산출물 | 포함할 내용 | 사용 목적 |
|--------|------------|----------|
| 04_metrics.json | | |
| 05_analysis.py | | |
| 05_analysis.ipynb | | |
| 06_dashboard.html | | |

## 제외 범위
- 현재 데이터로 하지 않을 분석과 이유

## 검증 계획
- 재현성 검증:
- 수치 검증:
- 해석 리스크:
```

## 원칙
- 계획은 짧고 실행 가능해야 한다.
- 답할 수 없는 질문을 억지로 넣지 않는다.
- 방법론 이름보다 성공 기준을 명확히 한다.
- 모든 산출물은 기본 한글로 작성한다.
